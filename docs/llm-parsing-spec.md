# LLM Parsing Spec

> Defines the Claude API contract for extracting structured invoice data from natural language voice/text input.

---

## Overview

The LLM parser is the core intelligence of Inavoice. It receives transcribed voice input (or raw text) and outputs a structured JSON object representing the invoice fields. It operates in two modes: initial parsing and correction parsing.

---

## LLM Output JSON Schema

```json
{
  "client_id": "string | null",
  "description": "string | null",
  "line_items": [
    {
      "service_date": "string (DD/MM/YYYY)",
      "service_description": "string | null",
      "time_start": "string (HH:MM, 24h format) | null",
      "time_end": "string (HH:MM, 24h format) | null",
      "rate": "number | null",
      "rate_type": "hourly | flat",
      "total": "number | null"
    }
  ],
  "missing_fields": ["string"]
}
```

### Field Definitions

| Field | Type | Source | Rules |
|-------|------|--------|-------|
| `client_id` | string or null | Matched from voice input against known client list | If unrecognized, set to null and add to missing_fields |
| `description` | string or null | Dictated by user OR pulled from client default | Header line above invoice table (e.g. "Invoice for AER Hong Kong, Art.berdeen booking") |
| `line_items` | array | Parsed from voice input | MVP: always exactly 1 item. Schema supports multiple. |
| `line_items[].service_date` | string | Dictated by user | Format: DD/MM/YYYY. If user says "March 26" → "26/03/2026". Use current year (HKT) if not specified. |
| `line_items[].service_description` | string or null | Dictated or from client default | Table cell text (e.g. "DJ set at AER (Aesthetic Radio) Hong Kong") |
| `line_items[].time_start` | string or null | Dictated by user | 24h format. "10 PM" → "22:00". "midnight" → "00:00". Null if rate_type is flat. |
| `line_items[].time_end` | string or null | Dictated by user | 24h format. Same conversion rules. Null if rate_type is flat. |
| `line_items[].rate` | number or null | Dictated or from client default | Per-hour rate if hourly, total fee if flat. Null triggers missing_fields. |
| `line_items[].rate_type` | string | Inferred from voice input | "hourly" if user says "per hour", "an hour", etc. "flat" if user says "flat fee", "agreed fee", or states a total without time. |
| `line_items[].total` | number or null | Computed by backend (not LLM) | LLM should set this to null. Backend computes: if hourly → rate × hours; if flat → rate. |
| `missing_fields` | array of strings | LLM-determined | List of field names the LLM could not extract. Bot will ask user for these. |

### Fields NOT in LLM output (computed/auto-generated by backend)

- `invoice_number` — auto-incremented from database
- `invoice_date` — always today (HKT)
- `due_date` — invoice_date + 14 days
- `hours` — computed from time_start and time_end
- `currency` — always HKD
- All sender details, bank details — static template data

## Client Matching

The LLM prompt includes the full list of known client display names plus any
spoken aliases stored on each contact. Aliases come from the
`contacts.aliases` column (comma-separated TEXT) and are rendered by
`_build_client_list` in `src/services/llm_parser.py` as ` (also: …)` after
the display name. Example:

```
Known clients:
- "client_a" → Client A Ltd. (also: AER, aesthetic)
- "client_b" → Client B Ltd.
```

### Matching Rules

1. The LLM maps spoken client references to a `client_id` from the known list.
2. The LLM should be tolerant of transcription errors (e.g. "a static radio" → "aesthetic_radio").
3. If the LLM cannot confidently match to a known client, it MUST set `client_id: null` and add `"client_id"` to `missing_fields`. It must NOT guess.
4. The backend validates that the returned `client_id` exists in the contacts database. If it doesn't (LLM hallucinated), treat as unrecognized.

---

## Missing Fields Handling

Required fields that trigger `missing_fields` if absent:

| Field | When required |
|-------|-------------|
| `client_id` | Always |
| `line_items[].service_date` | Always |
| `line_items[].rate` | When client has no `default_rate` |
| `description` | When client has no `default_description` |
| `line_items[].service_description` | When client has no `default_service_description` |
| `line_items[].time_start` / `time_end` | When `rate_type` is "hourly" |

When `missing_fields` is non-empty, the bot asks the user for each missing field before showing the confirmation.

## Mode 1: Initial Parsing

### System Prompt Structure

```
You are an invoice data extractor. Given a transcribed voice command, extract structured invoice data.

Today's date: {current_date_hkt} (timezone: HKT, Asia/Hong_Kong)

Known clients:
{client_list_with_aliases}

Rules:
- Output ONLY valid JSON matching the schema below. No explanation, no markdown.
- Match the spoken client name to a known client_id. If unsure, set client_id to null.
- Convert all times to 24-hour format (HH:MM).
- Convert all dates to DD/MM/YYYY format. If no year is stated, use the current year.
- If the user mentions "per hour" or "an hour", set rate_type to "hourly".
- If the user mentions "flat fee" or just states a total, set rate_type to "flat".
- Set total to null — the backend will compute it.
- List any fields you could not extract in missing_fields.
- Do NOT guess or hallucinate values. If unsure, add the field to missing_fields.

Output schema:
{json_schema}
```

### Example

**Input:** "Invoice client a, March 26, playing from 10 to midnight. Agreed fee was 500 Hong Kong dollars an hour."

**Output:**
```json
{
  "client_id": "client_a",
  "description": null,
  "line_items": [
    {
      "service_date": "26/03/2026",
      "service_description": null,
      "time_start": "22:00",
      "time_end": "00:00",
      "rate": 500,
      "rate_type": "hourly",
      "total": null
    }
  ],
  "missing_fields": []
}
```

Note: `description` and `service_description` are null here — the backend fills them from client defaults.

## Mode 2: Correction Parsing

When the user taps [Edit] and sends a correction, the LLM receives the previous parsed JSON plus the correction text.

### System Prompt Structure (Correction Mode)

```
You are an invoice data corrector. The user is editing a previously parsed invoice.

Previous parsed data:
{previous_json}

The user wants to make a correction. Apply their change to the previous data and output the complete updated JSON. Do not discard unchanged fields.

Rules:
- Output ONLY the complete updated JSON. No explanation.
- Only change the fields the user explicitly mentions.
- If the user's correction is ambiguous (e.g. "change the client" without saying which), keep the previous value and add the ambiguous field to missing_fields.
- Set total to null — the backend will recompute it.

Output schema:
{json_schema}
```

### Correction Scenarios

**Scenario A — single field:**
User says: "Rate should be 600"
LLM updates `rate: 600`, keeps everything else, sets `total: null`.

**Scenario B — multiple fields:**
User says: "It was March 28, and the time was 9 to 11 PM"
LLM updates `service_date: "28/03/2026"`, `time_start: "21:00"`, `time_end: "23:00"`, keeps everything else.

**Scenario C — ambiguous correction:**
User says: "Change the client"
LLM keeps previous `client_id`, adds `"client_id"` to `missing_fields`. Bot asks: "Which client should this invoice be for?"

## Edge Cases

### Time Parsing
| User says | LLM outputs |
|-----------|------------|
| "10 to midnight" | time_start: "22:00", time_end: "00:00" |
| "10 PM to 2 AM" | time_start: "22:00", time_end: "02:00" |
| "9 to 11" (ambiguous AM/PM) | Assume PM for typical service work. time_start: "21:00", time_end: "23:00" |
| "noon to 3" | time_start: "12:00", time_end: "15:00" |
| "midnight to 3 AM" | time_start: "00:00", time_end: "03:00" |

### Date Parsing
| User says | LLM outputs |
|-----------|------------|
| "March 26" | "26/03/{current_year}" |
| "26th of March" | "26/03/{current_year}" |
| "last Friday" | Compute actual date relative to today (HKT) |
| "yesterday" | Compute actual date relative to today (HKT) |
| No date mentioned | Add "service_date" to missing_fields |

### Rate Parsing
| User says | LLM outputs |
|-----------|------------|
| "500 an hour" | rate: 500, rate_type: "hourly" |
| "500 per hour" | rate: 500, rate_type: "hourly" |
| "flat fee of 2000" | rate: 2000, rate_type: "flat" |
| "2000 total" | rate: 2000, rate_type: "flat" |
| "agreed fee was 500 Hong Kong dollars an hour" | rate: 500, rate_type: "hourly" |
| No rate mentioned, client has default_rate | LLM sets rate: null, backend fills from default |
| No rate mentioned, client has no default_rate | rate: null, add "rate" to missing_fields |

## Validation Layer (Backend, post-LLM)

After the LLM returns JSON, the backend validates:

| Check | Action on failure |
|-------|------------------|
| `client_id` exists in contacts table | Treat as unrecognized client, ask user |
| `rate` > 0 (if provided) | Ask user to re-state rate |
| `service_date` is not > 90 days in the future | Ask user to confirm date |
| `service_date` is a valid date | Ask user to re-state date |
| `time_start` and `time_end` are valid HH:MM | Ask user to re-state time |
| JSON is parseable | Log error, ask user to try again |
| `rate_type` is "hourly" or "flat" | Default to "hourly" |

---

## Backend Computations (post-validation)

After validation passes, the backend computes:

1. **hours**: If `rate_type` is "hourly", compute from `time_start` and `time_end`. Handle midnight crossing (e.g. 22:00 → 02:00 = 4 hours).
2. **total**: If "hourly" → `rate × hours`. If "flat" → `total = rate`.
3. **invoice_date**: Current date in HKT.
4. **due_date**: `invoice_date + 14 days`.
5. **invoice_number**: Claimed via atomic upsert on invoice_counter table.
6. **description**: If null, pull from client's `default_description`.
7. **service_description**: If null, pull from client's `default_service_description`.
